# PRD Linear Planning

## When to use
- Use this as the canonical planner contract for SDLC planning in Linear.
- REQUIRE hierarchy: **Initiative = PRD**, **Project = User Story**, **Issue = HLD**.
- REQUIRE MCP state discovery at the start of every iteration.
- LLDs are NOT created here — they are generated by the sdlc-architect during execution.

## When NOT to use
- DENY use for implementation-only coding tasks.
- DENY use when only a standalone local document is requested with no Linear synchronization.
- DENY writes when MCP state cannot be confirmed.

## Inputs required
1. Idea/problem statement.
2. Desired outcome and constraints.
3. Relevant team/context (if known).

## Contract terms
- **REQUIRE**: mandatory condition that must be satisfied.
- **DENY**: action that is forbidden.
- **ALLOW**: action that is permitted only within stated bounds.

## Workflow

1. **MCP discovery gate (REQUIRE before any proposal/update)**
   - REQUIRE fresh MCP state snapshot for Initiative/Project/Issue scope.
   - DENY proposing or applying updates before this snapshot is captured.

2. **State assessment**
   - REQUIRE summary of current state, gaps, conflicts, duplicates, and stale items.

3. **Plan proposal sequence**
   - REQUIRE PRD refinement first using [`references/PRD.md`](references/PRD.md).
   - REQUIRE User Story decomposition second using [`references/USER-STORY.md`](references/USER-STORY.md).
   - REQUIRE HLD issue shaping third using [`references/HLD.md`](references/HLD.md).

4. **PRD validation gate (REQUIRE before decomposition)**
   - REQUIRE passing all 8 validation dimensions at "high" confidence before creating User Story Projects or HLD Issues.
   - DENY decomposition when any dimension is "low" or "medium" without explicit user risk acknowledgment.
   - See [`references/VALIDATION.md`](references/VALIDATION.md) for the full rubric, scoring criteria, and probing questions.

5. **Cross-level validation gates (REQUIRE)**
   - REQUIRE traceability links Initiative → Project → Issue.
   - REQUIRE overlap/conflict check across sibling Projects and sibling Issues.
   - REQUIRE measurable acceptance criteria at each level.

6. **Apply sequence**
   - REQUIRE apply order: Initiative → Project (User Story) → Issue (HLD).
   - DENY out-of-order writes unless parent links already exist and are verified.

7. **Communication and change-tracking gate (REQUIRE)**
   - REQUIRE each meaningful change to be tracked through either:
     - Initiative/Project update body, or
     - HLD Issue comment thread.
   - REQUIRE the minimal contract from [`references/COMMUNICATION.md`](references/COMMUNICATION.md): what changed, why, impact, next action.
   - REQUIRE use of Resources for durable context links (specs, decision docs, references) when applicable.

8. **MCP unavailable stop behavior**
   - DENY writes when MCP is unavailable or state fetch fails.
   - REQUIRE explicit blocker report with missing capability and impact.
   - ALLOW provisional local draft only when clearly marked `PROVISIONAL - NOT SYNCED TO LINEAR`.

9. **Iteration close**
   - REQUIRE compact per-iteration output contract (below).

## Output per iteration
1. **State snapshot**: MCP entities read + key IDs/statuses.
2. **Proposed changes**: planned Initiative/Project/Issue deltas and rationale.
3. **Applied changes**: confirmed writes (or explicit `none` when blocked).
4. **Communication trail**: update/comment references recorded for this iteration (or explicit `none` + reason).
5. **Next iteration focus**: smallest unresolved risk/gap to address next.

## Files
- [`references/PRD.md`](references/PRD.md): Initiative/PRD 14-section template and gates.
- [`references/USER-STORY.md`](references/USER-STORY.md): Project/User Story decomposition and overlap controls.
- [`references/HLD.md`](references/HLD.md): Issue/HLD readiness, acceptance, and traceability gates.
- [`references/VALIDATION.md`](references/VALIDATION.md): 8-dimension PRD validation rubric, scoring criteria, and probing questions.
- [`references/COMMUNICATION.md`](references/COMMUNICATION.md): Resources + Initiative/Project updates + HLD Issue comment guidance for auditable change tracking.

## Troubleshooting
- If state is stale/incomplete, REQUIRE re-fetch before any proposal or write.
- If MCP is unavailable, DENY writes and emit blocker + optional provisional local draft label.
- If Projects overlap, REQUIRE explicit boundary rewrite before creating/updating HLD Issues.
- If HLD Issues are vague, REQUIRE rewrite to explicit outcome + testable acceptance criteria.
- If PRD validation dimension is blocked on technical feasibility, DENY guessing — dispatch a research agent first.
